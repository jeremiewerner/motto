<objective>
Deliver `motto build` — the packager that turns a lint-passing source tree into self-contained, distributable Claude Code plugin folders under `dist/`. Build is lint-first (reuses Phase 2's `lintProject` as a gate) and copy-only (no content transform). It wipes `dist/`, copies each skill verbatim into `dist/<audience>/<name>/`, bundles declared `shared_references` into each skill's `references/`, and emits `dist/<audience>/.claude-plugin/plugin.json` per bucket.

Purpose: This is the final v1 phase — after it, `motto lint` + `motto build` are both real and a project ships standard Agent Skill plugins.
Output: `src/build.js` (`buildProject`), `test/build.test.js`, wired `bin/motto.js` build branch, `.gitignore` for `dist/`.
</objective>

## Phase Goal

**As a** skill author, **I want to** run `motto build` on a lint-passing project, **so that** I get self-contained, distributable Claude Code plugin folders ready for installation.

> Note: the ROADMAP `**Goal:**` line ("Developers can run `motto build` to produce self-contained, distributable Claude Code plugin folders ready for installation") is faithfully transcribed into the three user-story slots above — actor, capability, outcome — with no invented content.

## Acceptance Criteria (5 ROADMAP success criteria)
- [ ] SC1: lint-passing project → exit 0, reports output dir, each skill verbatim at `dist/<audience>/<name>/SKILL.md`, skills siblings of `.claude-plugin/`
- [ ] SC2: each `shared_references` entry copied into `dist/<audience>/<name>/references/<ref>.md`
- [ ] SC3: each bucket has `.claude-plugin/plugin.json` with `name`, `version`, `description`; `version` always present
- [ ] SC4: private bucket emitted only when ≥1 `audience:private` skill AND `plugins.private` set; else only `dist/public/`
- [ ] SC5: lint-failing project → exit 1, same lint diagnostics, writes nothing to `dist/`

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI argv → dispatch | The subcommand string `'build'` is used only for branch dispatch, never in path construction |
| source tree (skills/, shared/) → dist/ | Build reads validated source files and writes only under `join(projectRoot,'dist')` |
| motto.yaml / SKILL.md frontmatter → packaging | Plugin names + shared_reference names flow into file paths; all pre-validated by the Phase 2 lint gate |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Tampering | `dist/` wipe (`fs.rm`) | medium | mitigate | Wipe target is exactly `join(projectRoot,'dist')`; `fs.rm` removes only the symlink (not an external target) when `dist/` is a symlink — verified in RESEARCH. No user-controlled segment in the wipe path. |
| T-03-02 | Tampering / Information Disclosure | shared_reference path → `references/<ref>.md` copy | medium | mitigate | Build trusts the Phase 2 LINT-05 safe-basename contract (no `/` or `.` in ref names); names come from lint-passing frontmatter only. Build reads `shared/references/<ref>.md` under projectRoot exclusively. |
| T-03-03 | Tampering | verbatim symlink copy | low | accept | `verbatimSymlinks:true` copies an absolute symlink verbatim if a skill contains one. Correct behavior for a verbatim copy tool; no symlink-safety lint rule exists in REQUIREMENTS (out of scope). |
| T-03-SC | Tampering | npm/pip/cargo installs | low | accept | No new packages introduced this phase — all dependencies are Node stdlib + already-approved `yaml` (RESEARCH Package Legitimacy Audit). No install task, no legitimacy checkpoint required. |

ASVS L1 / block-on-high: no high-severity threats; no blocking checkpoint required.
