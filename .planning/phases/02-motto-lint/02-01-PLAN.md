<objective>
Deliver a working `motto lint` command: the filesystem + orchestration + CLI layer that wraps the Phase 1 pure validators (`parseFrontmatter`, `validateSkill`, `loadConfig`) into a runnable binary. Running `motto lint` against a project produces a complete, deterministic error report or a clean exit.

Purpose: Phase 1 built the pure validation core but deferred all I/O. This phase adds the discovery, error aggregation, and command surface that turn those functions into a tool a developer can actually run. This is the first user-facing capability of Motto.

Output: `src/lint.js` (lintProject orchestrator), `bin/motto.js` (thin CLI shell), `test/lint.test.js` (behavior suite), and a `bin` field in `package.json`.
</objective>

## Phase Goal (user story)

**As a** skill author, **I want to** run `motto lint` in my project, **so that** I get a complete, deterministic error report (or a clean pass) before I ship.

This phase is planned as a single vertical slice: a failing end-to-end test (Task 1) defines the behavior, the orchestrator (Task 2) makes it pass, and the CLI wiring (Task 3) lets a real user invoke it. After Task 3 a developer can run `motto lint` and see real output — a capability they did not have after Task 2.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem → lint process | `motto lint` reads `motto.yaml`, `skills/*/SKILL.md`, and `shared/references/*.md` from the author's own working tree. These are author-controlled, not third-party inputs. |
| CLI argv → process | The subcommand name (`positionals[0]`) and any flags cross from the shell into the process. Used only for dispatch — never in path construction. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-01 | Tampering | path construction in src/lint.js (`join(skillsDir, dirName, 'SKILL.md')`) | low | mitigate | `dirName` originates only from `readdir` output (OS-controlled), never from a user-supplied string; all paths are built with `path.join` rooted at `projectRoot` — no traversal vector. |
| T-02-02 | Tampering | shared_references resolution | low | mitigate | Phase 1 `validateSkill` already rejects entries containing `/` or `.` (safe-basename check, T-02-02 control); Phase 2 builds the sharedRefs Set only from real `shared/references/*.md` basenames, so no path escape is possible. |
| T-02-03 | Denial of Service | readdir over a huge skills/ tree or pathological YAML in a SKILL.md | low | accept | Local single-user dev tool operating on the author's own files; the author controls input size. Not worth bounding in v1 (YAGNI). |
| T-02-04 | Information Disclosure | error messages echo filesystem paths/filenames | low | accept | Greppable, path-naming diagnostics are a deliberate product feature (D2 discretion: errors must name the offending path). Paths belong to the author running the tool locally; no cross-tenant exposure. |
| T-02-SC | Tampering | npm/registry installs | low | accept | No new packages are introduced this phase — all dependencies are `node:*` stdlib plus the already-approved `yaml` (verified: RESEARCH Package Legitimacy Audit). No install task to gate. |
</threat_model>
