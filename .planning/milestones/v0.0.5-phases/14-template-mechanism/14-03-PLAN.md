<objective>
Close the single verification gap for Phase 14 (Roadmap SC3, TMPL-03): `hasClosedSection` (src/schema.js:63-80) produces two reproduced false positives. (1) Its fence scanner toggles `inFence` on ANY 3+ backtick-or-tilde line without tracking which character opened the fence or its length, so a body mixing `~~~` and ``` fences can count tags that are genuinely inside an open fence as real section markers. (2) It tests open- and close-tag presence independently with no ordering check, so a reversed `</process>` … `<process>` body registers as a valid closed pair. This plan fixes both defects in `hasClosedSection` and adds the two missing regression tests.

Purpose: Make SC3's "no false positives" and TMPL-03's "matched pair" guarantees hold on author-reachable input, restoring the mechanism's advertised rigor.
Output: Corrected `hasClosedSection` in src/schema.js and two new regression cases in test/schema.test.js.
</objective>

    Rewrite the internals of `hasClosedSection` in src/schema.js to close both defects. Keep the signature `hasClosedSection(body, tagName)`, the `export`, and the JSDoc `@param`/`@returns` contract unchanged. Keep the two-phase structure (line scan to strip fenced content, then anchored-regex match on the collected text).

    Fence tracking (WR-02): replace the boolean `inFence` with a nullable fence descriptor holding the opening fence's character and length. Keep the existing marker regex `/^ {0,3}(`{3,}|~{3,})/` but capture the run so you can derive `ch` (the first character of the captured run — a backtick or a tilde) and `len` (the run's length). On each marker line: if no fence is currently open, record the descriptor `{ ch, len }` and treat the line as a fence boundary (do not collect it). If a fence IS open, only close it (clear the descriptor) when the marker line's character equals the open fence's `ch` AND its length is >= the open fence's `len`; that closing line is a boundary (do not collect it). Any other marker line while a fence is open is literal fenced content — do not collect it and do not toggle. Every non-marker line is collected only when no fence is currently open. Net effect: a `~~~`-opened fence is closed only by a later `~~~` line of equal-or-greater length; a stray ``` line inside it is content, so tags inside the tilde fence stay excluded (fixes the mixed-fence false positive).

    Ordering (WR-01): after joining the collected unfenced lines into `text`, use `exec` (not `test`) for both the open and close anchored regexes (`^<tag>` and `^</tag>`, multiline). Return `true` only when both produce a match AND the open match's index is strictly less than the close match's index. A close-before-open document yields `false`. Preserve the existing open-tag semantics: no end-of-line anchor on the open matcher (trailing same-line content after `<process>` still counts, per Assumption A2 and the line-511 test).
