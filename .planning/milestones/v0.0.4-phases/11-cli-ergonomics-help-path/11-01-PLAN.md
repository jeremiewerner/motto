<objective>
Add discoverable usage text and directory-targeting to the `motto` CLI. After this plan a user who has never read the source can run `motto --help` (or `-h`, or bare `motto`) to see every command, run `motto <cmd> --help` to see one command's focused usage, and run `motto lint`/`motto build` against any directory (defaulting to cwd) with clear errors for bad paths and unknown input.

Purpose: Close CLIX-03 (help on demand, exit 0) and CLIX-04 (`[path]` targeting) — the two requirements that make Motto usable without reverse-engineering `bin/motto.js`.

Output: A modified `bin/motto.js` (help system + `[path]` wiring + directory guard) and a new `test/cli.test.js` that spawns the CLI as a child process and asserts stdout/stderr/exit-code behavior for every case.
</objective>

| Boundary | Description |
|----------|-------------|
| user argv → CLI (`bin/motto.js`) | Untrusted command-line input: subcommand token, `-h`/`--help`/`--force` flags, and the optional `[path]` positional. |
| CLI → filesystem (via `lintProject`/`buildProject` + the D-06 stat guard) | The `[path]` argument selects which directory Motto reads (skills) and writes (`dist/`) under. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-11-01 | Information Disclosure | `[path]` positional → lintProject/buildProject root | low | accept | Local single-user dev CLI operated on the user's own machine; user-supplied path is the intended CLIX-04 feature. lint only reads under the given root; build writes only `dist/` under it. No privilege boundary is crossed, so an arbitrary path grants the invoking user nothing they lack already. |
| T-11-02 | Denial of Service | argv parsing + `[path]` handling in `bin/motto.js` | low | mitigate | parseArgs `strict:true` rejects unknown flags (Task 1 catch); the Task 2 pre-dispatch stat guard rejects nonexistent/non-directory paths with `✗ directory not found` before any orchestrator work, preventing opaque downstream failures. The guard uses a non-throwing stat so malformed paths surface a message, never a stack trace. |
| T-11-SC | Tampering | npm/pip/cargo installs | low | accept | No new dependencies are added this phase. All new code uses Node stdlib only (`node:util` parseArgs already present; `node:child_process` + `node:test` in tests). No package-install task exists, so there is no supply-chain surface to gate. |

No threat is rated high or critical (ASVS level 1, blocking threshold "high"), so no blocking human checkpoint is required.
