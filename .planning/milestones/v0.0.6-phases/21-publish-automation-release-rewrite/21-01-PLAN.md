<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| tag push → CI publish job | A `v*` tag ref (write-access-gated) crosses into an authenticated CI job holding NPM_TOKEN |
| CI publish job → npm registry | Authenticated external write (`npm publish` via NODE_AUTH_TOKEN) |
| CI publish job → GitHub Releases API | Authenticated external write (`gh release create` via GITHUB_TOKEN) |
| repo working tree → npm tarball | `npm pack` output — files crossing the package boundary into what npm ships (D-05) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-21-01 | Information Disclosure / Tampering | NPM_TOKEN in publish job env | high | mitigate | Token injected only via `NODE_AUTH_TOKEN` into the single `npm publish` step; no `set -x`/verbose flags on any publish step; npm does not echo the token by default. (Token creation + granular scoping handled in 21-03.) |
| T-21-02 | Elevation of Privilege | GITHUB_TOKEN scope on `gh release create` | medium | mitigate | Job-level `permissions: { contents: write }` scoped to `publish` only; workflow default stays `contents: read`; `id-token: write` NOT granted this phase. |
| T-21-03 | Tampering | Idempotency-guard bug causing double / partial publish | medium | mitigate | Two independent guards (`npm_guard` / `gh_guard`), each on its own output; npm registry version-immutability is the backstop on the npm side; `gh release view` guard is the independent defense on the Releases side. |
| T-21-04 | Tampering | File leaking into the published tarball (secrets, planning, tests) | high | mitigate | `assertTarballClean` D-05 gate runs in `pack-install-e2e` on every push; adversarial unit test proves it throws on a leak (not a no-op). |
| T-21-05 | Elevation of Privilege / Repudiation | Anyone with repo write access pushes a `v*` tag → unintended publish | low | accept | Repo is private with a single maintainer holding write access (verified this milestone); blast radius limited to the maintainer's own actions; revisit tag-protection rules at the Phase 22 public flip. |
| T-21-SC | Tampering | npm/pip/cargo installs | low | accept | Zero new packages this phase (RESEARCH.md Package Legitimacy Audit: N/A); `npm ci` installs only already-locked dependencies from package-lock.json. |
</threat_model>
